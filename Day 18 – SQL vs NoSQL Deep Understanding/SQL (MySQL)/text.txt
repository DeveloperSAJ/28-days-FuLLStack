# 1️⃣ What Is Firebase?

Firebase is **Backend-as-a-Service (BaaS)** by Google.

It gives you:

* Database
* Authentication
* Hosting
* Cloud functions
* Storage
* Analytics
* Push notifications

👉 You don’t manage servers. Google does.

Perfect for:

* Startups
* MVPs
* Mobile apps
* Real-time apps

# 3️⃣ Firestore (Most Important)

Full name:
👉 Cloud Firestore

### 🔹 Type:

NoSQL Document Database

### 🔹 Structure:

```
Collection
   └── Document
          └── Fields
          └── Subcollections
```

Example:

```
users (collection)
   └── userId123 (document)
          ├── name: "Ali"
          ├── email: "ali@gmail.com"
          └── orders (subcollection)
                └── order1
```

Think of it like:

* JSON storage
* But structured into collections

---

## 🔹 How Data Is Stored

Example document:

```json
{
  "name": "Ali",
  "age": 22,
  "isActive": true,
  "createdAt": Timestamp
}
```

Firestore supports:

* String
* Number
* Boolean
* Timestamp
* Array
* Map (nested object)

---

# 4️⃣ Firestore vs Realtime Database

Older database:
👉 Firebase Realtime Database

| Feature     | Firestore             | Realtime DB          |
| ----------- | --------------------- | -------------------- |
| Structure   | Collections/Documents | Big JSON tree        |
| Scaling     | Better                | Limited              |
| Queries     | More powerful         | Basic                |
| Recommended | ✅ Yes                 | Only for simple apps |

👉 For modern apps → Use Firestore.

# 5️⃣ How To Use Firestore (Step-by-Step)

### Step 1: Create Firebase Project

* Go to Firebase Console
* Create project
* Add web/app

---

### Step 2: Install SDK (Web Example)

```bash
npm install firebase
```

---

### Step 3: Initialize

```javascript
import { initializeApp } from "firebase/app";
import { getFirestore } from "firebase/firestore";

const firebaseConfig = {
  apiKey: "...",
  authDomain: "...",
  projectId: "..."
};

const app = initializeApp(firebaseConfig);
const db = getFirestore(app);
```

---

### 🔹 Add Data

```javascript
import { collection, addDoc } from "firebase/firestore";

await addDoc(collection(db, "users"), {
  name: "Ali",
  age: 22
});
```

---

### 🔹 Get Data

```javascript
import { getDocs } from "firebase/firestore";

const querySnapshot = await getDocs(collection(db, "users"));
querySnapshot.forEach((doc) => {
  console.log(doc.id, doc.data());
});
```

---

### 🔹 Query Data

```javascript
import { query, where } from "firebase/firestore";

const q = query(
  collection(db, "users"),
  where("age", ">", 20)
);
```

---

### 🔥 Real-Time Listener

```javascript
import { onSnapshot } from "firebase/firestore";

onSnapshot(collection(db, "users"), (snapshot) => {
  snapshot.forEach((doc) => {
    console.log(doc.data());
  });
});
```

This updates automatically when data changes.

That’s Firebase power.

# 6️⃣ Authentication (Very Important)

Firebase Auth supports:

* Email/password

```javascript

const auth = getAuth();

```

# 7️⃣ Security Rules (CRITICAL)

This is where many beginners fail.

Example rule:

```js
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    match /users/{userId} {
      allow read, write: if request.auth != null;
    }
  }
}
```

Meaning:

* Only authenticated users can access.

You can restrict:

* Only owner can edit
* Only admin can delete
* Role-based access

# 🔟 Cloud Storage

For:

* Images
* Videos
* Files

Example:

```javascript
import { getStorage, ref, uploadBytes } from "firebase/storage";

const storage = getStorage();
const storageRef = ref(storage, 'images/photo.jpg');
uploadBytes(storageRef, file);
```

# 1️⃣1️⃣ How Firebase Is Different From Traditional Backend

Traditional:
Frontend → Backend → Database

Firebase:
Frontend → Firebase directly

No custom backend needed (unless advanced logic).

You would use:

# 1️⃣3️⃣ When To Use Firebase?

Use it when:

* You need fast MVP
* You want real-time updates
* You don’t want to manage servers
* Small team
* Mobile app

---

# 1️⃣4️⃣ When NOT To Use Firebase?

Avoid when:

* Extremely complex relational queries
* Heavy backend logic
* Very strict SQL-based reporting
* Large enterprise systems
